# 블로그 작성 기준

블로그 글을 쓸 때 따르는 기본 기준을 정리한다.  
글감이 기술 설명인지, 회고인지, 후기인지에 따라 자연스러운 말투가 다르기 때문에 카테고리별 기준을 분리해서 가져간다.

## 1. 기본 원칙

1. 한 글 안에서는 말투를 섞지 않는다.
2. 기술 설명 글은 독자에게 설명하는 문서처럼 쓴다.
3. 회고나 후기는 내 경험을 정리하는 기록처럼 쓴다.
4. 제목은 검색되기 쉬운 표현을 우선한다.
5. `description`은 글을 클릭하기 전에 무엇을 얻을 수 있는지 바로 알 수 있게 쓴다.

## 2. 카테고리별 말투 기준

### 존댓말로 쓰는 카테고리

- `Next.js`
- `React`
- `CSS`
- `Django`
- `Project`
- 구현/문제 해결 중심의 `기타`

이 카테고리들은 설명글 성격이 강하므로 기본적으로 존댓말로 쓴다.  
문장 끝은 `정리했습니다`, `설명합니다`, `사용할 수 있습니다`, `확인하시면 됩니다`처럼 마무리한다.

### 반말로 쓰는 카테고리

- `회고`
- `컨퍼런스`
- `책`
- 후기 성격의 `기타`

이 카테고리들은 기록과 감상에 가깝기 때문에 반말이 더 자연스럽다.  
문장 끝은 `했다`, `좋았다`, `느꼈다`, `돌아본다`처럼 마무리한다.

## 3. description 작성 기준

`description`은 메타데이터이지만 사실상 글의 한 줄 소개다.  
그래서 제목의 반복보다는, 이 글이 무엇을 다루는지와 어떤 정보를 얻을 수 있는지를 바로 보여주는 문장으로 쓴다.

### 기술 설명 글의 description

- `구현 과정을 정리했습니다`
- `주의할 점을 정리했습니다`
- `문제 해결 과정을 정리했습니다`
- `비교한 내용을 정리했습니다`

예시:

- `React Hook Form을 적용해 회원가입 폼을 구현해본 과정을 정리했습니다.`
- `PWA 사이트를 구글 플레이 스토어에 출시한 과정과 겪은 이슈를 정리했습니다.`

### 회고/후기 글의 description

- `느낀 점을 돌아본다`
- `준비 과정을 돌아본다`
- `배운 점을 정리한다`
- `참가 후기를 남긴다`

예시:

- `AWS SAA-C03를 준비하고 취득하면서 느낀 점을 돌아본다.`
- `인프콘 2024에 다녀오며 들은 세션과 느낀 점을 돌아본다.`

## 4. 도입부 작성 기준

### 기술 설명 글

도입부에서는 아래 순서를 기본으로 잡는다.

1. 어떤 상황에서 이 작업을 하게 되었는지
2. 왜 이 주제가 필요했는지
3. 이 글에서 무엇을 정리할 것인지

예시 흐름:

`프로젝트를 진행하면서 이 문제를 직접 다루게 되었습니다. 처음에는 단순해 보였지만 실제로는 여러 조건을 함께 고려해야 했습니다. 이 글에서는 구현 과정에서 주의했던 점들을 정리해보겠습니다.`

### 회고/후기 글

도입부에서는 배경과 감정을 먼저 둔다.

1. 어떤 계기였는지
2. 왜 의미 있었는지
3. 이 글에서 무엇을 돌아볼 것인지

예시 흐름:

`올해는 유난히 변화가 많았다. 예상하지 못한 일도 많았고, 그만큼 많이 배웠다. 이 글에서는 그 과정을 돌아보려고 한다.`

## 5. 기술 글 본문 작성 기준

1. 가능하면 번호를 붙여 읽기 쉽게 쓴다.
2. `문제 상황 -> 왜 필요한지 -> 코드 -> 주의할 점` 순서를 우선한다.
3. 단정할 수 없는 내용은 가능성 표현으로 쓴다.
4. workaround는 workaround라고 적는다.
5. 브라우저 정책이나 플랫폼 제약은 코드로 완전히 해결된다고 쓰지 않는다.

## 6. 회고/후기 글 본문 작성 기준

1. 시간 순서나 주제 순서로 자연스럽게 쓴다.
2. 감상만 쓰지 말고, 무엇을 배웠는지까지 남긴다.
3. 과장된 교훈보다는 구체적인 경험을 우선한다.
4. 지나치게 설명체로 바꾸지 않는다.

## 7. 앞으로의 정리 기준

1. 새 글을 쓸 때는 먼저 카테고리를 정한다.
2. 카테고리를 정한 뒤 말투를 먼저 결정한다.
3. `description`은 마지막에 쓰되, 제목 반복이 아니라 글의 효용을 요약한다.
4. 기존 글은 수정이 필요할 때 이 기준에 맞춰 점진적으로 정리한다.
