비디오 자동재생 인터렉션을 구현할 때는 단순히 `video` 태그에 영상만 연결한다고 끝나지 않습니다. 브라우저 정책, iPhone 동작 방식, 첫 프레임 처리, 폴백 이미지, 투명 비디오 대응까지 함께 고려해야 실제 서비스에서 안정적으로 동작합니다.

이번 글에서는 제가 비디오 자동재생 인터렉션을 구현할 때 주의하는 점들을 순서대로 정리해보았습니다. 특히 랜딩 페이지나 프로모션 페이지처럼 짧은 반복 영상, 배경 영상, 인터랙션 영상이 들어가는 상황을 기준으로 작성했습니다.

## 1. `autoplay`와 `muted`를 함께 붙입니다

자동재생 비디오를 구현할 때 가장 먼저 확인할 것은 `autoplay`와 `muted`입니다.

```tsx
<video autoPlay muted />
```

WebKit 정책 기준으로도 음성이 없거나 `muted`인 경우에 한해 자동재생이 허용됩니다. 그래서 `autoplay`만 넣고 `muted`를 넣지 않으면 실제 기기에서 자동재생이 막히는 경우가 자주 발생합니다.

### 넣지 않았을 때 생길 수 있는 문제

1. 어떤 브라우저에서는 자동재생이 아예 되지 않을 수 있습니다.
2. 기기나 브라우저 정책에 따라 동작이 들쭉날쭉해질 수 있습니다.
3. 초기 화면에서 영상이 보여야 하는데 정지된 상태로 남을 수 있습니다.

## 2. iPhone 이슈를 줄이기 위해 `playsInline`을 추가합니다

iPhone에서는 `playsinline`이 없는 `video`가 전체 화면으로 전환될 수 있습니다. 자동재생 배경 영상이나 작은 인터랙션 영상에서는 이런 동작이 UX를 크게 깨뜨립니다.

```tsx
<video autoPlay muted playsInline />
```

인라인 자동재생을 의도하신다면 `playsInline`은 사실상 같이 들어간다고 보시는 편이 안전합니다.

## 3. 저전력 모드에서는 자동재생이 실패할 수 있습니다

이 부분은 코드로 완전히 해결할 수 있는 영역이 아닙니다. iPhone의 저전력 모드나 브라우저 정책에 따라 자동재생이 실패하는 사례가 실제로 보고되어 있습니다.

이런 상황에서는 "자동재생은 항상 성공하지 않을 수 있다"는 전제로 설계하시는 편이 맞습니다.

1. 자동재생이 실패해도 레이아웃이 깨지지 않아야 합니다.
2. 영상 대신 보여줄 `poster`나 이미지 폴백을 준비해두어야 합니다.

## 4. 빈출 이슈 대응으로 `#t=0.001`을 시도해볼 수 있습니다

iOS Safari에서 첫 프레임이 비어 보이거나, 시작 지점 처리가 어색한 경우 우회책으로 URL 뒤에 `#t=0.001` 같은 media fragment를 붙여보는 경우가 있습니다.

```tsx
<source src="/videos/landing-video.mp4#t=0.001" type="video/mp4" />
```

이 방식은 표준 속성이 아니라 media fragment를 이용해 재생 시작 지점을 아주 조금 뒤로 미루는 방법입니다. 따라서 만능 해결책으로 쓰기보다는, 특정 환경에서만 유효할 수 있는 우회 방법으로 이해하시는 편이 맞습니다.

## 5. `poster` 속성으로 기본 폴백을 넣어둡니다

`poster`는 비디오가 재생되기 전 보여줄 이미지를 지정하는 속성입니다.

```tsx
<video
  autoPlay
  muted
  playsInline
  loop
  poster="/images/landing-video-poster.png"
>
  <source src="/videos/landing-video.mp4" type="video/mp4" />
</video>
```

자동재생 비디오에서는 첫 진입 순간이 중요합니다. 영상 로딩이 늦어지거나 자동재생이 바로 시작되지 않으면 검은 화면이나 빈 영역처럼 보일 수 있는데, 이럴 때 `poster`를 넣어두면 초기 화면을 조금 더 자연스럽게 만들 수 있습니다.

다만 `poster`는 어디까지나 재생 전 상태를 위한 이미지이므로, 재생 실패 상황 전체를 책임지는 폴백으로 보기는 어렵습니다.

## 6. `video` 바깥에 이미지 레이어를 폴백으로 둡니다.

`poster`만으로는 초기 로딩이나 재생 실패 상황을 항상 안정적으로 커버하기 어렵기 때문에,

`video` 아래에 이미지를 먼저 깔아두고 그 위에 비디오를 겹쳐 올리는 방식을 사용하면 좋습니다.

Next.js에서는 아래처럼 구현할 수 있습니다.

```tsx
'use client';

import Image from 'next/image';

export default function LandingVideo() {
  return (
    <div style={{ position: 'relative', aspectRatio: '16 / 9' }}>
      <Image
        src="/images/landing-video-poster.png"
        alt=""
        fill
        priority
        sizes="100vw"
        style={{ objectFit: 'cover' }}
      />

      <video
        autoPlay
        muted
        playsInline
        loop
        poster="/images/landing-video-poster.png"
        style={{
          position: 'absolute',
          inset: 0,
          width: '100%',
          height: '100%',
          objectFit: 'cover',
        }}
      >
        <source src="/videos/landing-video.webm" type="video/webm" />
        <source src="/videos/landing-video.mp4#t=0.001" type="video/mp4" />
      </video>
    </div>
  );
}
```

## 7. 경우에 따라서는 이미지 시퀀스 기법을 사용할 수도 있습니다

강한 인터랙션이 필요할 때는 실제 비디오 대신 여러 장의 이미지 프레임을 순서대로 보여주는 이미지 시퀀스 방식을 사용하기도 합니다.

이미지 시퀀스는 `video` 태그로 영상을 재생하는 것 대신, 여러 장의 프레임 이미지를 준비한 뒤 JavaScript로 현재 프레임을 제어하면서 `img`나 `canvas`에 순서대로 보여주는 방식입니다.

장점은 특정 프레임을 정밀하게 제어하기 쉽다는 점입니다. 예를 들어 스크롤 위치에 따라 원하는 장면에서 정확히 멈추게 만들거나, 사용자의 인터랙션에 맞춰 프레임을 세밀하게 바꿀 수 있습니다.

반면 이미지 수가 많아지면 용량과 로딩 비용이 커질 수 있어서, 일반적인 배경 영상 대체재로는 항상 적합하지 않습니다.

## 8. non-visible 상태에서 발생할 수 있는 이슈가 있습니다

WebKit 정책상 `<video autoplay>` 요소는 화면에 보일 때 재생을 시작하고, non-visible 상태가 되면 일시정지될 수 있습니다.

그래서 iOS Safari에서는 자동재생 비디오가 화면 밖으로 나가거나 비가시 상태가 되면 멈추고, 다시 화면 안으로 들어왔을 때도 재생 복구가 매번 일관적이지 않을 수 있습니다.

이 부분은 브라우저 정책의 영향이 커서 완전히 제어하기 어렵습니다.

## 9. 투명 영상

배경이 투명한 영상을 웹에서 사용하려면 일반적인 mp4 하나만으로는 부족한 경우가 많습니다. 브라우저마다 지원 포맷과 코덱이 다르기 때문입니다.

보통은 아래처럼 두 가지 버전을 준비하게 됩니다.

1. 대부분의 브라우저용 `webm`
2. Safari 대응용 `mp4` 또는 `mov`

예를 들면 다음과 같이 구성할 수 있습니다.

```tsx
<video autoPlay muted playsInline loop poster="/images/poster.png">
  <source src="/videos/landing-video.webm" type="video/webm" />
  <source
    src="/videos/landing-video.mov#t=0.001"
    type='video/mp4; codecs="hvc1"'
  />
</video>
```

### 9-1. `webm`과 `mp4` 또는 `mov`를 같이 준비해야 할 수 있습니다

Chrome 계열과 Safari 계열이 동일하게 처리되지 않기 때문에, 투명 비디오라면 하나의 포맷만으로는 대응이 어려울 수 있습니다.

### 9-2. `source` 순서가 중요합니다

브라우저는 위에서부터 재생 가능한 첫 번째 source를 사용합니다. 그래서 보통은 `webm`을 먼저 두고, Safari용 소스를 뒤에 둡니다.

### 9-3. 코덱을 명시해두는 편이 안전합니다

`type`에 코덱 정보를 명시해두면 브라우저가 지원하지 않는 코덱을 더 정확하게 건너뛰고 다음 source를 선택하는 데 도움이 됩니다.
명시하지 않으면 안드로이드에서 mov 파일을 재생하여 검은 화면이 나오는 이슈가 발생할 수 있습니다.

예를 들어 HEVC 기반이라면 아래처럼 적을 수 있습니다.

```tsx
<source
  src="/videos/landing-video.mov#t=0.001"
  type='video/mp4; codecs="hvc1"'
/>
```


## 10. 알아두면 좋은 `ffmpeg` 활용 방법

`ffmpeg`는 영상과 오디오를 변환, 편집, 분석할 때 널리 쓰이는 오픈소스 멀티미디어 CLI 도구입니다. 영상 압축, 썸네일 추출, 코덱 확인, 포맷 변환 등에 활용할 수 있기 때문에 알아두면 좋습니다.


### 10-1. 설치

macOS에서는 Homebrew로 바로 설치할 수 있습니다.

```bash
brew install ffmpeg
```

코덱 확인이 필요하다면 `ffprobe`도 함께 사용하시면 됩니다.

### 10-2. 썸네일 추출

poster나 폴백 이미지를 만들 때 가장 자주 쓰는 명령어입니다.

```bash
ffmpeg -ss 0.001 -i landing-video.mp4 -vframes 1 -q:v 2 landing-video-poster.png
```

### 10-3. 코덱 확인

확장자만 보지 말고 실제 코덱을 확인하시는 편이 좋습니다.

```bash
ffprobe -v error -select_streams v:0 -show_entries stream=codec_name,codec_tag_string -of default=noprint_wrappers=1 your-video.mov
```

예를 들어 아래처럼 나오면:

```bash
codec_name=hevc
codec_tag_string=hvc1
```

HEVC(H.265) 기반 영상이라고 보시면 됩니다.

### 10-4. 포맷 변환

기본적인 webm -> mp4 변환 예시는 아래처럼 사용할 수 있습니다.

```bash
ffmpeg -i input.webm -c:v libx264 -crf 23 -c:a aac -movflags +faststart output.mp4
```

### 10-5. 영상 압축 또는 크기 줄이기

자동재생 비디오는 용량이 조금만 커져도 체감 성능에 바로 영향을 줍니다. 그래서 해상도나 프레임 수를 줄이는 작업도 자주 하게 됩니다.

```bash
ffmpeg -i input.mov \
-vf "scale=400:-1:flags=lanczos" \
-r 20 \
-c:v prores_ks \
-profile:v 4444 \
-qscale:v 26 \
-pix_fmt yuva444p10le \
output.mov
```

## 마무리

비디오 자동재생 인터렉션은 보기에는 단순하지만, 실제로는 브라우저 정책과 기기별 제약을 많이 받는 요소입니다.
모든 케이스를 대응하는것은 힘들기 때문에 실패 케이스를 같이 설계해두는 것이 좋습니다.
